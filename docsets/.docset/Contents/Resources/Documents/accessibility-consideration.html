Accessibility

As the goal is to make this feature work for visually impaired people we need to take into consideration the accessibility aspect.

Below are some things to consider while developing the UI as the SDK does not provide any accessibility functionality out of the box.

How to use the VoiceOver

A short introduction on to how to use the functionality on iOS to get you started with testing.

You move in lists while swiping with 2 fingers to right or left to move up or down the list.

You click something when having the button focused and one finger on the screen and tap with the second finger to confirm.

Use swipe with 3 fingers up or down to scroll.

Font size

There are people who are using a magnifing glass and read the text letter by letter.

Therefore a larger font size should be used and line breaks within a word should be avoided at all cost

Contrast

A higher contrast can help people navigate better and identify areas of interest.

This could be for example a white play button on a black background.

The Image of the play button should also be a lot bigger.

Screenreader

You can easily test your application with the screenreader enabled in iOS.

Things to consider for screenreader:

  • Are there visual elements with no information? Disable them for the screenreader in UIKit you would do this with isAccessibilityElement = false

  • Add information to any icon that displays a state - e.g. a play button displaying the play icon. You can do this in UIKit with accessibilityLabel

  • A text has changed on the screen? You can refocus it to force the screenreader to read it again. This would be done in UIKit with Â UIAccessibility.post(notification: .screenChanged, argument: THE_LABEL)

  • Keep the focus in the overlay if displayed. If you display a bottom sheet or a popup dialog you should use a focus trap (if not provided by the system) so that the user cannot navigate out of the element with the screenreader

  • Use the correct elements or at least provide the correct roles. If you have an element that you can click and it is not a button you need to provide the correct traits. In UIKit it would be accessibilityTraits . Why? Because the screenreader would read first the text of the button and afterwards ‘Button’ or ‘Taste’ in german. This is mandatory for the user to know, that this element is clickable.

  • Everything above will have something similar in SwiftUI

Announcer

Sometimes it makes sense to announce something for the user (spoken by the system).

Below are some consideration when to use this functionality.

  • Something changed in the UI state without the interaction of the user. For example a new StreamGroup was added. You could announce that information (but maybe only when the user is not activly listening to another stream)

  • A state has changed e.g. the Stream that the user is currently listening too has ended and was removed.
